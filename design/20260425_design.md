# Full Spec: Rust Runner-Based ReAct LLM Agent + Text UI for Colossal Cave

Build a Rust system that:

1. Uses the local `runner` crate to run Colossal Cave (`games/advent.z5`)
2. Sends one game command at a time through `runner::Runner`
3. Captures and cleans game output returned by the runner
4. Maintains transcript + structured world memory
5. Uses a ReAct LLM agent via OpenRouter and ChatGPT 5.4
6. Provides an interactive Text UI (TUI)
7. Allows both manual play and autonomous agent task execution
8. Allows task interruption (`/cancel`)
9. Uses free text input as LLM task requests
10. Exits cleanly with `exit` or `/exit`


The program must not implement or duplicate the interpreter bridge. Game execution must go through the `runner` crate public API.

The system must be fully implemented with all features in this spec.

---

# 6. Runner API Used by This Program

This program needs the following runner-facing operations:

```rust
use runner::{CommandResult, Runner, RunnerError};
```

```rust
let mut runner = Runner::load_story(&config.game.story_path)?;
let output = runner.send_command("look")?;
```

The app may also use `run_commands` for tests and scripted smoke checks. Autonomous play and manual TUI input should use `send_command` so each turn produces one observation.

The runner crate owns interpreter initialization, command injection, output buffering, native integration, and shutdown. Those details are outside this program's design.

---

# 7. Output Handling

The runner returns raw text output for each command. The application is responsible for preparing that text for display, transcript storage, and LLM prompts.

Required cleanup:

* normalize line endings to `\n`
* trim trailing prompt artifacts only when they are not meaningful game text
* preserve meaningful spacing inside room descriptions and inventory lists
* remove terminal control sequences if any appear
* keep the unmodified command string in the transcript alongside the cleaned observation

---

# 8. Game Command Execution

All game commands must pass through a single `GameSession`-style wrapper around `runner::Runner`.

```rust
pub struct GameSession {
    runner: runner::Runner,
}

impl GameSession {
    pub fn load(story_path: &str) -> Result<Self, GameError>;

    pub fn execute(&mut self, command: &str) -> Result<Observation, GameError>;
}
```

`execute` must:

* receive an already validated single-line command
* call `Runner::send_command`
* clean the returned output
* append command and observation to the transcript
* return the observation to the UI and agent layers

---

# 9. Runner Error Handling

Runner failures must be converted to application-level errors near the game session layer.

Rules:

* story load failure prevents startup and shows a clear system error
* command failure records the failed command and displays a system message
* output-too-large or invalid UTF-8 errors stop the active agent task
* `exit` or `/exit` should drop the `GameSession` after transcript and memory persistence complete

The UI and agent layers should not branch on runner internals.

# 19. Input Rules

Always-active input field.

---

## Rule A - Exit

```text
exit
/exit
```

Actions:

* stop active task
* save transcript + memory
* drop the runner-backed game session
* exit cleanly

---

## Rule B - Cancel

```text
/cancel
```

Actions:

* immediately stop current LLM task
* do not stop game
* return to waiting-for-input state

---

## Rule C - Direct Game Command

Examples:

```text
/send look
/send north
/send take lamp
/send inventory
```

Strip the `/send` prefix, trim the remaining text, validate it as one game command, then execute it directly through `GameSession`, which calls `runner::Runner::send_command`.

No LLM used.

---

## Rule D - Free Text = LLM Task

Example:

```text
What can be done with the bird in the cage?
```

Start autonomous ReAct task.

Only one active task allowed.

If one already exists:

```text
A task is already running. Use /cancel first.
```
