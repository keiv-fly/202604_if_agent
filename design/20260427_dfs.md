# DFS Command Planner Design

Date: 2026-04-27

## Goal

Add a deterministic DFS-based command planner that can generate the next command plan from:

* current DFS state
* current structured world information
* the current frontier set
* latest game observation and transcript context

The planner must be usable by the `eval` binary so evaluation can explore the early Colossal Cave map without asking the LLM for every move. The planner receives the full frontier set, selects one frontier action, and proposes a list of commands: navigation commands needed to reach the location where the selected frontier action can be executed, followed by the selected frontier action itself. The design must still allow the LLM to be triggered for specific events while that list is being executed, inspect the situation, and modify the DFS state, move list, frontier set, selected action, route, or world information.

This document is a design specification only. It does not require source changes yet.

## Current Baseline

The existing eval loop in `src/bin/eval.rs` does this:

1. load game session
2. execute initial `look`
3. update `WorldModel`
4. repeatedly ask the LLM for one game command
5. optionally normalize movement-only commands
6. execute the command
7. update `WorldModel`
8. score discovered locations against `eval_data/first_nodes.json`

The current world model in `src/memory/world.rs` tracks:

* `current_location`
* `locations`
* `inventory`
* `important_objects`
* `hypotheses`
* `task_notes`

Locations currently track:

* `title`
* `description`
* `exits`
* `objects`
* `notes`

The new DFS planner should sit between the agent/eval loop and `GameSession::execute`. In eval mode, it becomes the primary source of route plans. The executor still sends exactly one command at a time to the game. The LLM becomes an optional advisory or repair mechanism that can react between commands in the active plan.

## Non-Goals

The initial DFS planner should not:

* solve puzzles
* execute non-movement actions by default
* replace the `WorldModel`
* depend on runner internals
* require a TUI redesign
* make hidden assumptions from `first_nodes.json`
* use the ground-truth eval data to choose moves

The first implementation should be movement-focused exploration.

## High-Level Flow

The eval loop should become:

1. load game session
2. execute initial `look`
3. update world model
4. initialize DFS planner state from current world
5. repeatedly ask the planner to choose from the frontier set and produce a route to the selected action
6. validate the returned command list before execution starts
7. execute the plan one command at a time
8. after each command, classify the observation and update world/DFS state
9. after each command, decide whether the remaining plan is still valid
10. optionally trigger LLM analysis for configured events
11. apply any accepted LLM patches to DFS/frontier/world/active-plan state
12. continue, replan, or stop based on the updated state

The planner is responsible for choosing a command list. The game session remains responsible for executing one validated command at a time. The active-plan executor is responsible for stopping between commands when the world diverges from the plan.

## Core Concepts

### DFS State

The DFS state represents the planner's traversal memory, separate from but linked to `WorldModel`.

It should contain:

* current node id or location key
* stack of traversal frames
* path from root to current location
* visited location keys
* attempted exits per location
* failed exits per location
* frontier set
* discovered edges
* known routes between locations, when available
* active command plan, when available
* active plan cursor
* turn counters and safety limits
* last command
* last observation classification

A traversal frame should represent one location in the active DFS path and the movement options still available from that location.

The DFS state should be serializable so eval logs can include planner state snapshots.

### World Information

The `WorldModel` remains the source of structured observations about the game world.

The DFS planner should read:

* current location title
* current location description
* known exits
* known destination for each exit, when available
* known objects and notes, when available

The planner may propose world-model updates, but direct mutation should happen through a controlled integration layer. This avoids mixing traversal decisions with observation parsing.

### Frontier Set

At planning time the planner receives a frontier set. Each frontier action represents a known but not fully explored opportunity that DFS may choose to execute.

Each frontier action should contain:

* stable action id
* source location key
* direction or command to try
* expected destination, if known
* status: pending, routing, executing, explored, failed, blocked, needs-analysis
* priority
* reason the entry exists
* discovery turn
* last attempted turn

The frontier set is not the same as the DFS stack:

* DFS stack describes the current depth-first traversal path.
* The frontier set describes all known exploration actions that have not been fully resolved.
* The selected frontier action describes the single exploration action targeted by the returned command plan.
* The route plan describes how to get from the current location to the selected frontier action's source location.

The planner should normally prefer DFS order, but because it receives the whole frontier set it may choose a different pending action when the first candidate is unreachable, blocked, lower priority, or requires analysis. The selection rule must be deterministic and logged.

The planner should not silently discard unreachable actions. If one action is skipped in favor of another, the decision should record the skipped action id and reason.

## Location Identity

The planner needs stable location keys. The first implementation should use a conservative key derived from:

* normalized location title
* normalized description
* visit order disambiguator when title and description are ambiguous

Title-only identity is insufficient because Colossal Cave can contain repeated or similar room titles.

The location key should be internal to the planner/world integration. Eval scoring should continue to use the existing title and title-description comparison.

## Movement Commands

The initial movement command list should be centralized and shared with eval normalization.

Canonical movement commands:

* `north`
* `south`
* `east`
* `west`
* `northeast`
* `northwest`
* `southeast`
* `southwest`
* `up`
* `down`
* `in`
* `out`

Aliases may be accepted at input boundaries, but planner output should use canonical commands only.

The planner should expose the move list as editable state so the LLM can reorder, remove, add, or annotate moves for specific situations. The route planner should use the known edge graph and reverse-direction table to produce navigation commands; it should use the canonical move list when creating new frontier actions in the frontier set.

## Command Planner Contract

Create a function whose responsibility is:

Given current DFS state, current world information, the frontier set, and latest observation context, return the next command-plan decision.

The command plan is a list of commands with this shape:

1. zero or more route commands that should move the player from the current location to the selected frontier action's source location
2. exactly one selected frontier action command to execute at that source location

If the player is already at the frontier source location, the list contains only the frontier action command.

The decision should distinguish:

* execute a command plan
* request LLM analysis
* stop because exploration is complete
* stop because safety limits were reached
* stop because planner state is inconsistent

The function should not execute any command itself. It only chooses one frontier action from the frontier set and returns the full list of commands needed to reach and execute that action.

The function should be deterministic for the same inputs unless explicitly configured otherwise.

The returned plan should be treated as provisional. After every command execution, the integration layer must check whether the remaining commands are still valid for the observed current location. If not, it should discard or suspend the remaining plan and request replanning or analysis.

## Planner Decisions

The planner should return a structured decision with enough information for logs and eval output.

Decision types:

* `CommandPlan`: execute a validated list of game commands, one command per game turn
* `Analyze`: trigger LLM with a specific event and structured context
* `Complete`: no useful frontier actions remain
* `Blocked`: cannot safely route to or execute any eligible frontier action
* `Error`: planner state is invalid

For command-plan decisions, include:

* ordered command list
* route commands
* frontier action command
* start location key
* frontier source location key
* expected location after each route command, when known
* selected frontier action id
* skipped frontier action ids and reasons, when applicable
* reason
* whether each command is route movement, backtracking, or frontier execution

The command list should not be sent to the game as a batch. It is an execution plan that the eval loop consumes one command at a time.

For analysis decisions, include:

* event type
* human-readable reason
* relevant DFS state slice
* frontier set slice
* selected frontier action, if any
* active command plan and cursor, if any
* remaining commands in the plan
* relevant world slice
* allowed patch types

## Active Plan Execution

The active plan executor owns the process of consuming a `CommandPlan`.

Execution rules:

* validate every command in the plan before execution starts
* execute only the next command, never the full list as a game command batch
* update world and DFS state after every command
* classify each observation before advancing the plan cursor
* verify that the observed location matches the expected route location when one is known
* pause the plan if an LLM trigger event fires
* discard the remaining plan if the current location no longer matches the route
* mark the selected frontier action as attempted only when the frontier action command is executed

This lets the deterministic planner work without an LLM today while leaving a clear insertion point for future LLM reactions between route steps.

## Observation Classification

After each executed command, the integration layer should classify the result before updating DFS/frontier state.

Useful classifications:

* moved to known location
* moved to new location
* command failed or direction blocked
* command had no location change
* ambiguous location change
* game ended or fatal state
* observation parse failed
* route step matched expected location
* route step diverged from expected location
* frontier action executed

This classification does not need to be perfect. It should be conservative and trigger LLM analysis when uncertainty matters.

## DFS Update Rules

After a successful route move:

* advance the active plan cursor
* update current location
* refresh known edge information
* continue executing the remaining route only if the observed location matches the plan

After a route move reaches an unexpected location:

* pause or discard the remaining active plan
* record the unexpected edge or ambiguity
* mark the selected frontier action as still pending unless the frontier command already ran
* trigger analysis or replanning

After the frontier action successfully moves to a new location:

* mark the selected frontier action as explored
* record the edge from source to destination
* push destination onto DFS stack
* add new pending actions to the frontier set for the destination using the current move list
* mark destination visited

After the frontier action successfully moves to a known location:

* record or refresh the edge
* mark the selected frontier action as explored
* if this move was planned backtracking, pop or adjust the DFS stack
* if this move was forward exploration, decide whether the known destination should be revisited or skipped

After a failed route command:

* pause or discard the remaining active plan
* mark the route edge as failed or non-reversible, depending on context
* keep the selected frontier action pending
* trigger replanning or analysis

After a failed frontier action command:

* mark that direction as failed or blocked for the source location
* mark the selected frontier action as failed
* keep current DFS stack unchanged

After an ambiguous result:

* pause the active plan
* mark the selected frontier action or active route step as needs-analysis
* trigger LLM analysis unless ambiguity is explicitly configured to be ignored

## Backtracking

Backtracking should prefer known reverse edges.

Reverse direction table:

* `north` -> `south`
* `south` -> `north`
* `east` -> `west`
* `west` -> `east`
* `northeast` -> `southwest`
* `southwest` -> `northeast`
* `northwest` -> `southeast`
* `southeast` -> `northwest`
* `up` -> `down`
* `down` -> `up`
* `in` -> `out`
* `out` -> `in`

Some game maps are not perfectly reversible. If a reverse command fails or lands somewhere unexpected while building or executing a route plan, the planner should mark the edge as non-reversible and trigger analysis or path repair.

The route planner should first try to find a known path from the current location to each eligible frontier action's source location, then choose the best reachable action according to deterministic DFS/frontier priority. Acceptable path sources are:

* direct known forward edges
* known reverse edges that have not been marked non-reversible
* DFS stack ancestry, when it is consistent with current location

If no path is known for any eligible frontier action, the planner should not fabricate route commands. It should return `Blocked` or `Analyze`.

## LLM Trigger Events

The design should allow LLM analysis to be triggered for specific events instead of every turn.

Initial trigger events:

* observation could not be classified
* movement command appears blocked but text suggests an alternate exit
* location identity is ambiguous
* route step produced an unexpected location
* route to any eligible frontier action cannot be found
* backtracking or reverse edge failed
* DFS stack and world current location disagree
* active command plan became invalid
* frontier set is empty but world notes suggest unexplored exits
* repeated command guard would stop the run
* max consecutive failed moves reached
* planner reaches a named special location that requires policy input
* eval run completes and needs a final world/frontier consistency review

Trigger policy should be configurable for eval:

* never call LLM
* call LLM only on repair events
* call LLM on repair events and frontier-set exhaustion
* call LLM every N turns for audit

The initial eval integration should default to repair-only or no-LLM mode, depending on the experiment being run.

## LLM Analysis Contract

When triggered, the LLM should receive structured context:

* event type
* event reason
* latest command
* latest observation
* current location
* compact transcript tail
* relevant world model slice
* DFS stack
* frontier set slice
* selected frontier action, if any
* active command plan
* remaining route commands
* failed or ambiguous exits
* allowed patch operations

The LLM should not directly execute a command. It should return proposed patches and optionally a recommended next planner decision or replacement command plan.

Allowed patch operations:

* add frontier action
* remove or disable frontier action
* reprioritize frontier action
* mark frontier action status
* replace selected frontier action
* replace frontier set
* replace or truncate active command plan
* request replanning from current location
* add or update edge
* mark edge non-reversible
* rename or merge location keys
* update current location key
* add location note
* add world hypothesis
* reorder move list
* disable a move at a specific location
* request stop with reason

Patch application should be validated before mutating planner or world state.

## Eval Integration

The eval binary should gain a DFS mode.

Recommended CLI options:

* `--strategy llm`: current behavior
* `--strategy dfs`: deterministic DFS planner
* `--strategy dfs-llm`: DFS planner with event-triggered LLM repair
* `--dfs-max-turns <N>`: planner-specific turn cap
* `--llm-trigger-policy <POLICY>`: controls LLM event triggers
* `--planner-log <PATH>`: optional planner trace output

The existing `--command` option can remain for LLM strategy. For DFS strategy, it can be ignored, logged as an eval task label, or used as high-level policy text for LLM analysis events.

Eval output should include:

* selected strategy
* number of planner turns
* number of generated command plans
* number of LLM analysis calls
* number of executed movement commands
* number of failed moves
* number of discovered locations
* frontier counts by status
* selected frontier action id, when a plan is active
* current score fields

The existing score calculation should remain unchanged.

## Logging

Planner logs should be explicit enough to replay decisions.

Recommended log events:

* `planner_init`
* `planner_decision`
* `planner_command_plan`
* `planner_command`
* `planner_plan_paused`
* `planner_replan`
* `planner_observation_classification`
* `planner_frontier_update`
* `planner_dfs_update`
* `planner_llm_trigger`
* `planner_llm_patch`
* `planner_stop`

Each decision log should include:

* turn
* current location key
* command, command-plan id, or decision type
* reason
* selected frontier action id, when applicable
* skipped frontier action ids, when applicable
* active plan cursor, when applicable
* remaining plan length, when applicable
* DFS stack depth
* frontier counts by status

## Safety Rules

The planner must preserve the existing command safety rules:

* exactly one command per turn
* no command batching
* command must pass `validate_game_command`
* every command in a returned plan must pass `validate_game_command`
* movement-only DFS mode emits canonical movement commands only
* no use of eval ground truth for planning
* stop on max-turn limit
* stop or trigger repair on repeated invalid state transitions

The planner should have its own guards:

* max turns
* max failed moves per location
* max total failed moves
* max backtracking attempts for the same edge
* max LLM repair attempts for the same event
* max command-plan length
* max replans for the same selected frontier action
* max frontier set size

## Suggested Module Shape

Add a new planner module under `src/planner/`.

Suggested responsibilities:

* DFS state data structures
* frontier set and frontier action data structures
* movement command constants and reverse-direction helpers
* command-plan decision function
* active-plan executor helpers
* route-finding helpers over known edges
* observation classification helpers
* state update helpers
* LLM event and patch schemas

The eval binary should depend on this module instead of duplicating DFS logic locally.

If the planner later becomes useful in the TUI, `src/agent` can call the same planner rather than creating a second exploration implementation.

## Testing Plan

Unit tests should cover:

* default frontier creation from a new location
* command-plan creation for a frontier action at the current location
* command-plan creation for a frontier action at another known location
* selecting among multiple reachable frontier actions
* skipping unreachable frontier actions without deleting them
* route command ordering
* failed move handling
* successful new-location move handling
* successful known-location move handling
* active-plan cursor advancement
* active-plan invalidation on unexpected location
* backtracking command selection
* non-reversible edge handling
* frontier set exhaustion
* location-key disambiguation
* LLM trigger event generation
* patch validation

Integration tests should cover:

* a short scripted fake world where DFS visits all rooms
* a fake world with blocked directions
* a fake world with one non-reversible edge
* a fake world where the selected frontier action requires multiple route commands first
* a fake world with multiple frontier actions where the nearest DFS-eligible action is selected
* a fake world where route execution diverges and forces replanning
* eval `--strategy dfs` completing without LLM calls
* eval `--strategy dfs-llm` triggering a mocked repair event

## Open Questions

* Should the planner add all canonical movement commands to every new location, or should it rely only on exits extracted from observations once exit extraction exists?
* Should `WorldModel::Location::exits` be populated by deterministic parsing, LLM patches, or both?
* Should planner state be saved alongside `memory_store/world-state.json` after eval runs?
* Should `dfs-llm` patches be applied automatically when valid, or should high-risk patches require stricter policy checks?
* Should the first implementation keep DFS state local to eval, or immediately expose it for TUI reuse?
* Should active plans be fully prevalidated against known route expectations, or should the executor allow unknown expected locations for early exploration?
* Should route commands and frontier action commands share one sequence type with metadata, or use separate fields in the command-plan decision?
* Should frontier action selection be strict DFS stack order, nearest reachable action, or DFS priority with route cost as a tiebreaker?

## Initial Implementation Recommendation

Implement in two phases.

Phase 1:

* deterministic DFS planner
* canonical move list
* frontier set and DFS stack
* deterministic frontier action selection
* route generation to the selected frontier action source
* command-plan execution one command at a time
* conservative observation classification
* eval `--strategy dfs`
* planner logging
* no LLM patches yet

Phase 2:

* LLM trigger events
* LLM patch schema
* validated patch application
* replacement command plans from LLM analysis
* eval `--strategy dfs-llm`
* mocked tests for repair behavior

This keeps the first change measurable and avoids mixing the deterministic traversal core with LLM repair semantics too early.
